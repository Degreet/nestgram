Your bot speaks the user's language; your code shouldn't have to keep saying
which one. In most codebases i18n means threading a `lang` argument through
every function that produces text ‚Äî handler to service to formatter ‚Äî until
half your signatures carry it. Nestgram resolves the locale **once per
update** and makes a translator ambiently available to everything that runs
for that update.

:::mental
update -> resolve locale\* -> ambient store -> t() anywhere -> reply
:::

## Why `t()` is a free function

`t()` is an **imported function, not an injected service** ‚Äî deliberately. An
injected translator only solves half the problem: every class that produces
text needs the injection, and the _locale_ still has to travel as a method
argument from the handler into every helper that formats a string. That's
exactly the threading we're killing. Instead, the dispatcher runs each
update inside an `AsyncLocalStorage` scope ‚Äî the same ambient rail
[sessions](/docs/sessions) ride. A built-in pipeline stage resolves the
sender's locale, seeds a locale-bound translator into that scope, and `t()`
reads it from wherever it's called in that update's call chain. The honest
trade-off: an ambient value is invisible in signatures, and
`AsyncLocalStorage` is in-process ‚Äî both bite in specific places, and this
page names them.

## Register the module

`I18nModule.forRoot()` takes your catalogs and a default locale. Import it
once, alongside `NestgramModule`:

:::code[app.module.ts]{mark="12"}

```ts
import { Module } from '@nestjs/common';
import { join } from 'path';
import { I18nModule, NestgramModule, directoryTranslations } from 'nestgram';

@Module({
  imports: [
    NestgramModule.forRoot({
      token: process.env.BOT_TOKEN ?? '',
      polling: true,
    }),
    I18nModule.forRoot({
      source: directoryTranslations(join(__dirname, 'locales')),
      defaultLocale: 'en',
    }),
  ],
})
export class AppModule {}
```

:::

Catalogs come from exactly one of three places: a `source` like
`directoryTranslations` (loaded once at boot), inline `translations` when a
TypeScript object suits you better, or a `backend` for a different message
format (Fluent, below). Configuring none ‚Äî or two ‚Äî fails at boot, not at
the first missed translation.

```ts
import { I18nModule, Translations } from 'nestgram';

const translations: Translations = {
  en: { 'start.greeting': 'Hi, {name}!' },
  uk: { 'start.greeting': '–ü—Ä–∏–≤—ñ—Ç, {name}!' },
};

I18nModule.forRoot({ translations, defaultLocale: 'en' });
```

:::note
Need catalogs from `ConfigService` or an async loader? Use
`I18nModule.forRootAsync({ useFactory, inject })`, exactly like other Nest
dynamic modules.
:::

## Locale files

With `directoryTranslations`, one file per locale: the filename is the
locale, the extension picks the parser ‚Äî `en.json`, `uk.yaml`, `de.yml`.
Keys nest naturally and flatten to dotted keys at load (`list.empty`);
`{name}` placeholders interpolate at translation time. An array value joins
with newlines ‚Äî handy for multi-line help text.

:::tabs{name="locales"}
::tab[locales/en.json]

```json
{
  "menu": { "remind": "‚ûï Remind", "help": "‚ÑπÔ∏è Help" },
  "start": { "greeting": "Hi, <b>{name}</b>!" },
  "list": {
    "empty": "No pending reminders üéâ",
    "header": "You have <b>{count}</b> pending reminder(s):"
  }
}
```

::tab[locales/uk.yaml]

```yaml
menu:
  remind: ‚ûï –ù–∞–≥–∞–¥–∞—Ç–∏
  help: ‚ÑπÔ∏è –î–æ–ø–æ–º–æ–≥–∞
start:
  greeting: –ü—Ä–∏–≤—ñ—Ç, <b>{name}</b>!
list:
  empty: –ù–µ–º–∞—î –Ω–∞–≥–∞–¥—É–≤–∞–Ω—å üéâ
  header: '–£ —Ç–µ–±–µ <b>{count}</b> –Ω–∞–≥–∞–¥—É–≤–∞–Ω—å:'
help:
  text: # an array joins into one multi-line message
    - <b>–ë–æ—Ç-–Ω–∞–≥–∞–¥—É–≤–∞—á</b> ‚Äî –ø—ñ–Ω–≥—É—é, –∫–æ–ª–∏ —á–∞—Å.
    - –ù–∞–¥—ñ—à–ª–∏ <code>10m –ü–æ–¥–∑–≤–æ–Ω–∏—Ç–∏ –º–∞–º—ñ</code>.
```

:::

:::caution
JSON works out of the box; YAML files need the optional `yaml` package
(`npm i yaml`) ‚Äî Nestgram tells you exactly that at boot if it's missing.
Two files for the same locale (`en.json` + `en.yaml`) are a boot error, not
a silent merge.
:::

## Use `t()` anywhere

Import `t` and call it ‚Äî in the handler, or in any service the handler
ends up calling. No locale in sight:

:::code[reminder.router.ts]{mark="8-9"}

```ts
import { Injectable } from '@nestjs/common';
import { Command, Message, Router, Sender, t, User } from 'nestgram';

@Injectable()
export class ReminderService {
  listText(pending: string[]): string {
    return pending.length === 0
      ? t('list.empty')
      : t('list.header', { count: pending.length });
  }
}

@Router()
export class ReminderRouter {
  constructor(private readonly reminders: ReminderService) {}

  @Command('start')
  start(message: Message, @Sender() user: User) {
    return t('start.greeting', { name: user.first_name });
  }

  @Command('list')
  list(message: Message) {
    return this.reminders.listText([]);
  }
}
```

:::

:::anno

1. `ReminderService` knows nothing about Telegram, locales or i18n config ‚Äî it calls `t()` and gets text in the right language, because it runs inside the same update's call chain.
2. The second argument is interpolation params: `{count}` in the template becomes the value you pass. An unmatched placeholder stays visible in the output (`{cuont}` is a bug you can see), it is never silently blanked.
3. A key missing from every catalog comes back **as the key itself** ‚Äî visible in chat, never a throw. A bot should still answer even when a translation is missing.

:::

Need to render a _different_ language inside the same update ‚Äî say, a
bilingual confirmation? Pass an explicit locale as the last argument:
`t('start.greeting', { name }, 'uk')`. This form still reads the per-update
translator from the ambient store, so it works anywhere in the update's call
chain ‚Äî but only there (the queue section below covers the rest).

:::warn[Nestgram warning]
[I18n] Missing i18n key "list.heder" for locale "uk" (and fallback) ‚Äî
returning the key verbatim.

> fix the key in the catalog ‚Äî and keep `logMissingKeys: true` on while
> developing to surface gaps like this (warned once per locale+key, off by default)

:::

## Localized buttons that still route

A reply keyboard built from `t()` renders in the sender's language ‚Äî which
means the button text that comes back differs per user, and a plain
`@Hears('‚ûï Remind')` only matches one of them. `@HearsKey()` is the
i18n-aware `@Hears`: it matches a message whose text equals the translation
of a **key** in the current update's locale. The locale is resolved before
matching runs, so this works as routing, not as an afterthought.

:::code[menu.router.ts]{mark="20"}

```ts
import { Command, HearsKey, Message, ReplyKeyboard, Router, t } from 'nestgram';

const MENU = {
  remind: 'menu.remind',
  help: 'menu.help',
} as const;

@Router()
export class MenuRouter {
  @Command('start')
  start(message: Message) {
    const keyboard = new ReplyKeyboard()
      .text(t(MENU.remind))
      .text(t(MENU.help))
      .resize();

    return message.answer(t('start.greeting'), { reply_markup: keyboard });
  }

  @HearsKey(MENU.remind)
  remind() {
    return t('remind.prompt');
  }

  @HearsKey(MENU.help)
  help() {
    return t('help.text');
  }
}
```

:::

The same `MENU.remind` constant builds the button and routes the press ‚Äî
one name, defined once, no per-locale predicates and no drifting literals.

## When you need the code itself

Sometimes you want the locale, not a translation ‚Äî picking a date format,
logging, storing a preference. `@Locale()` injects it into a handler;
the free `locale()` returns the same value anywhere in the call chain:

```ts
import { Command, Locale, Message, Router } from 'nestgram';

@Router()
export class LangRouter {
  @Command('lang')
  lang(message: Message, @Locale() lang: string) {
    return `Resolved locale: ${lang}`;
  }
}
```

With `I18nModule` configured the resolved locale is always a string (your
`defaultLocale` at worst). Without it, `locale()` returns `undefined` ‚Äî like
everything here, it degrades instead of throwing.

## How the locale is resolved

Per update, in this order:

1. **Your `resolveLocale(ctx)`**, if you configured one ‚Äî else the default:
   the sender's Telegram `language_code` (what their client reports).
2. The candidate is kept **only if a catalog exists for it** ‚Äî a German user
   with only `en`/`uk` catalogs gets `defaultLocale`, not raw keys.
3. No candidate at all (channel posts have no sender; `language_code` is
   optional) ‚Üí `defaultLocale`.

Then, at translation time, each key falls back independently: the resolved
locale's catalog ‚Üí the `fallbackLocale` catalog (defaults to
`defaultLocale`) ‚Üí the key itself. A half-translated catalog ships usable ‚Äî
untranslated keys read in your fallback language instead of breaking.

A custom resolver is just a function over the update's context:

```ts
I18nModule.forRoot({
  translations: { en: {}, uk: {} },
  defaultLocale: 'en',
  resolveLocale: (ctx) => {
    const code = ctx.from?.language_code;
    if (!code) {
      return undefined; // ‚Üí defaultLocale
    }
    const [base] = code.split('-'); // 'pt-br' ‚Üí 'pt'
    return base;
  },
});
```

:::note
The resolver is synchronous and runs **before** sessions load (built-in
stage order: i18n `10`, sessions `20`) ‚Äî so it can't read a session-stored
language preference. If users pick a language inside your bot, load that
preference in your own earlier `UpdateStage` (order below `10`) and read it
in `resolveLocale` from `ctx.state` ‚Äî the same public stage mechanism the
framework's i18n uses. See [Extending Nestgram](/docs/extending).
:::

## Plurals and selectors: the Fluent backend

Flat catalogs with `{name}` interpolation cover most bots; plurals are where
they stop ‚Äî "1 reminders" is the classic giveaway. For real plural rules
(and other in-message selectors) Nestgram ships a backend for
[Fluent](https://projectfluent.org), Mozilla's message format, as `.ftl`
files:

:::code[locales/en.ftl]

```ftl
pending = { $count ->
    [one] You have { $count } pending reminder
   *[other] You have { $count } pending reminders
}
```

:::

Register it as the `backend` ‚Äî `fluentDirectory` is async (it lazily loads
the parser and reads your files once), so use `forRootAsync`:

```ts
import { join } from 'path';
import { I18nModule, fluentDirectory } from 'nestgram';

I18nModule.forRootAsync({
  useFactory: async () => ({
    backend: await fluentDirectory(join(__dirname, 'locales')),
    defaultLocale: 'en',
  }),
});
```

Your calling code doesn't change: `t('pending', { count: 3 })`. There's also
`fluentBackend({ en: enFtl, uk: ukFtl })` for inline `.ftl` strings.

:::caution
Fluent is an optional dependency ‚Äî `npm i @fluent/bundle` ‚Äî loaded lazily,
so bots on flat catalogs don't carry it. One honest default we flipped:
Fluent normally wraps interpolated values in invisible Unicode bidi
isolation marks, which surprise in bot messages (they break `@Hears`-style
exact matching, for one). Nestgram turns that off; opt back in with
`{ useIsolating: true }` if you render genuinely mixed-direction text.
:::

## Across a queue or worker boundary

The honest caveat: `AsyncLocalStorage` is **in-process and per-call-chain**.
A BullMQ job runs in a different call chain ‚Äî often a different process ‚Äî
where the ambient store simply doesn't exist: `t()` returns the key,
`locale()` returns `undefined`. The framework doesn't pretend otherwise; you
carry the locale explicitly. Capture it into the job payload when you
enqueue, and rebuild a translator in the worker from `I18nService` (exported
by `I18nModule` for exactly this):

:::code[remind.router.ts]{mark="15"}

```ts
import { InjectQueue } from '@nestjs/bullmq';
import { Queue } from 'bullmq';
import { Command, Message, Router, locale } from 'nestgram';

@Router()
export class RemindRouter {
  constructor(@InjectQueue('reminders') private readonly queue: Queue) {}

  @Command('remind')
  async remind(message: Message) {
    // capture the ambient locale NOW ‚Äî it won't survive into the worker
    await this.queue.add('deliver', {
      chatId: message.chat.id,
      text: 'Buy milk',
      locale: locale() ?? 'en',
    });

    return 'Scheduled.';
  }
}
```

:::

:::code[reminder.processor.ts]{mark="22"}

```ts
import { Processor, WorkerHost } from '@nestjs/bullmq';
import { Job } from 'bullmq';
import { BotService, I18nService } from 'nestgram';

interface DeliverJob {
  chatId: number;
  text: string;
  locale: string;
}

@Processor('reminders')
export class ReminderProcessor extends WorkerHost {
  constructor(
    private readonly bot: BotService,
    private readonly i18n: I18nService,
  ) {
    super();
  }

  async process(job: Job<DeliverJob>): Promise<void> {
    // no ambient context here ‚Äî rebuild a translator from the carried locale
    const translate = this.i18n.translator(job.data.locale);
    await this.bot.sendMessage(
      job.data.chatId,
      translate('remind.delivered', { text: job.data.text }),
    );
  }
}
```

:::

:::caution
The explicit-locale form `t(key, locale)` does **not** help here ‚Äî it still
reads the per-update translator factory from the ambient store, so outside
an update it returns the key too. `I18nService.translator(locale)` is the
one form that works with no update in flight.
:::
